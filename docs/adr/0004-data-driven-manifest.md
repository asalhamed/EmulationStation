# ADR-0004: Data-driven system manifest

**Status:** Accepted (2026-05-17)

## Context
Upstream `prepare.ps1` repeats this pattern ~20 times, once per system:
```powershell
$nesCore = "$requirementsFolder\fceumm_libretro.dll.zip"
if(Test-Path $nesCore){
    Expand-Archive -Path $nesCore -Destination $coresPath | Out-Null
} else {
    Write-Host "ERROR: $nesCore not found."
    exit -1
}
```
And then the `es_systems.cfg` is generated as a ~250-line heredoc with `$variable` interpolation. Adding a system means editing both blocks; the script grows linearly with system count.

## Decision
- Every system is declared **once** in `manifest/systems.psd1` as a typed record.
- Install code is **generic** — iterates the manifest, doesn't name systems.
- `es_systems.cfg` is generated by rendering `templates/es_systems.cfg.template` over the manifest.

## Manifest schema (sketch)
```powershell
@{
  Name              = 'nes'
  FullName          = 'Nintendo Entertainment System'
  RomExtensions     = @('.nes', '.fds', '.unif', '.unf')
  Theme             = 'nes'
  Launcher          = @{
    Kind            = 'Libretro'                   # or 'Standalone'
    LibretroCore    = 'fceumm_libretro.dll'
  }
  WinGetPackages    = @('Libretro.RetroArch')
  HomebrewArtifact  = 'assimilate_full.zip'        # references downloads.psd1
  Notes             = 'Homebrew compilation by Wave 5'
}
```

## Why
- **Adding a system is one PR to the manifest.** Reviewable, mechanically simple, no code-path expansion.
- **The manifest is the spec.** `Get-EmulationStationManifest` outputs it; we can generate README tables from it; we can validate it without running anything.
- **Easier to test.** Generic `Install-System` cmdlet gets one set of tests; we don't write per-system tests.
- **Reflects how `es_systems.cfg` is structured anyway.** Each `<system>` element in ES's config is the same shape; the manifest mirrors it directly.

## Consequences
- A handful of systems need special handling that doesn't fit the generic shape (PS2's hacked-up `${ps2Binary}` path in upstream, Wii needing a `.wad` placed in the rom dir). We handle these via an optional `PostInstall` script block per system in the manifest — escape hatch with a documented contract.
- Manifest validation has to be strict — a typo in `LibretroCore` is silent until install. `Resolve-Manifest` enforces required fields, valid extension format, and existence of the referenced download/package.
- Schema migrations are a thing if we ever change the manifest shape. For v0.1 we accept this — there's only one consumer (us).

## What we explicitly don't do
- We don't try to be a general-purpose ES-system generator. Our manifest is opinionated to what we ship. If someone wants to add an obscure system, they edit the manifest in their fork.
- We don't read upstream's `es_systems.cfg` and try to import it — that file is an artifact, not a source of truth.
